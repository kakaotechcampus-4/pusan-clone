# Week 4 — 나나의 기억 검색 (retrieve nana's memory)

## 이번 주 목표
3주차까지 저장한 기억을 **출처별로 검색**해서 답변에 쓰게 한다. 핵심 메시지는 "RAG를 하나의 마법 함수로 보지 말고, 데이터 출처마다 검색 tool을 분리하라"는 것이다.

## 검색 출처 3종류 (기본은 앞 2개)
| tool | 대상 | 방식 |
|------|------|------|
| `search_personal_references` | 내가 적어둔 메모/참고자료 | ChromaDB 벡터 검색 (RAG) |
| `search_saved_requests` | 저장된 일정/할 일/알림 | SQLite 텍스트 조회 |
| `search_conversation_messages` (심화) | 과거 대화 발화 | ChromaDB, lazy sync |

## 배운 것

### 1. 모든 검색이 RAG는 아니다
일정 검색(`search_saved_requests`)은 SQLite 텍스트 조회이고, 참고자료 검색만 벡터 기반 RAG다. 정확한 값(날짜·이름)은 DB 조회가, 의미가 비슷한 자유 텍스트는 벡터 검색이 강하다. 출처 성격에 맞는 방식을 고르는 게 이번 과제의 핵심이었다.

### 2. 벡터DB는 만드는 게 아니라 갖다 쓴다
임베딩(텍스트→벡터)은 OpenAI 임베딩 API가, 저장·거리 계산은 ChromaDB가 담당한다. 내가 짠 tool은 store를 부르고 결과를 `json_payload`로 감싸는 얇은 입구다. 반환 키 계약(참고자료=`hits`, 일정=`rows`)을 지키는 게 중요했다.

### 3. 저장/검색 라우팅은 LLM이 프롬프트로 판단한다
사용자 입력이 일정인지 메모인지, 어느 저장소를 검색할지는 코드가 아니라 LLM이 프롬프트를 보고 고른다. 그래서 라우팅 기준을 프롬프트에 명확히 적어야 한다. 나는 "일정이 아니면 무조건 메모"가 아니라 "사용자가 기억/메모 의도를 밝혔을 때만 참고자료로 저장"하도록 정했다. 잡담까지 저장소에 쌓이는 것을 피하기 위해서다.

## 직접 확인한 한계 — RAG 할루시네이션
검색 관련성 판단이 코드가 아니라 LLM/프롬프트에 의존하다 보니, 결과가 일관되지 않았다.

- 저장한 것: `비밀번호 = 4590` (용도 미지정)
- 질문: "내가 저장한 현관 도어락 비번 뭐였지?"
- 답변: "저장된 현관 도어락 비밀번호는 4590입니다" ← "현관 도어락"은 저장한 적 없는데 단정 (할루시네이션)

반면 "이대리 연락처?"(김/박대리와 이름이 명확히 다름), "내 생일?"(되물음)은 잘 방어됐다. 같은 구조인데 라벨이 뭉개지면 단정하고 명확하면 방어한다.

방어 프롬프트("검색 결과가 없으면 지어내지 말고, 저장된 내용만 근거로 삼는다")를 추가하니 "검색 0건"은 막혔지만, 위 "현관 도어락"처럼 "관련은 있으나 질문과 정확히 일치하지 않는" 경우는 강한 문구를 넣어도 막지 못했다.

**결론**: 프롬프트 방어는 "검색 0건"에는 효과가 있지만 "애매한 부분 일치"에는 보장이 안 된다. 이 경계를 확실히 하려면 코드 단의 distance threshold, re-ranking, hybrid search 같은 보완이 필요하다. 이번 과제는 agentic RAG의 기초(LLM이 출처별 tool을 고르는 것)까지가 범위라, 프롬프트 방어를 불완전한 대안으로 두었다.
